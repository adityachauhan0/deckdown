# DeckDown v1.2.2 README Refresh Design

## Goal

Prepare the `v1.2.2` release surface with a fully rewritten `README.md`, a new two-slide showcase deck rendered by DeckDown itself, and fresh localhost Studio screenshots that prove the updated Studio workflow visually.

## Product Direction

- Position DeckDown as a polished product, not just a repo utility.
- Keep install and quickstart near the top so the README stays actionable for npm/GitHub users.
- Use a warm editorial visual language rather than a stark technical one.
- Show Studio as supporting proof, not as the sole product story.

## Approved Inputs

- Release version: `v1.2.2`
- Tagline: `Markdown to presentation engine, open source and local.`
- Tone: polished product
- Visual direction: warm editorial
- README priority: install and quickstart remain near the top
- Visual mix: rendered showcase deck plus fresh live Studio screenshots

## Scope

### In scope

- Replace `README.md` with a new version
- Create a new DeckDown showcase deck for the README visuals
- Render the showcase deck to PNG assets
- Capture new localhost Studio screenshots using the new deck
- Run the new deck locally in Studio so it can be reviewed on localhost

### Out of scope

- Large documentation restructure outside README
- Changes to CLI/API behavior unrelated to release assets
- New export formats or renderer features
- Packaging/version bump work beyond README/release-surface preparation

## Deliverables

### 1. New README

The new `README.md` should follow this arc:

1. Product title and tagline
2. Install
3. Quickstart
4. Short proof-oriented product summary
5. Showcase section using the new rendered deck and fresh Studio screenshots
6. Product capability sections:
   - repo-native authoring
   - local Studio workflow
   - Mermaid and LaTeX rendering
   - multi-output delivery
7. Docs/examples/release verification lower on the page

The README should feel more curated than the current one, but still be practical for first-time adopters.

### 2. New showcase deck

Create a new DeckDown deck specifically for the `v1.2.2` README visuals.

Slide 1 requirements:

- Artistic hero slide
- Warm editorial theme
- Introduce DeckDown
- Prominently feature the approved tagline:
  - `Markdown to presentation engine, open source and local.`
- Feel like a product launch slide, not a sample template

Slide 2 requirements:

- Pitch-style feature slide made in actual DeckDown syntax
- Show core proof points:
  - Studio workflow
  - Mermaid rendering
  - LaTeX/math rendering
  - multi-format output (`PDF`, `PNG`, `PPTX`)
  - repo-native authoring
- The slide must still feel like a real DeckDown presentation, not a static mockup image

### 3. Rendered assets

Render the new deck to PNG under `docs/assets/showcase/` or a nearby release-appropriate location already consistent with the repo structure.

The rendered assets should be the primary README visuals for the release deck itself.

### 4. Fresh Studio screenshots

Capture localhost Studio screenshots with the new deck open so the README also proves the current Studio surface. The screenshots should visibly show:

- file tree / repo-first workspace
- markdown editor
- rendered preview
- the modern Studio shell already built in this branch

## File Strategy

Expected file areas:

- `README.md`
- `samples/` for the new showcase deck source
- `docs/assets/showcase/` for generated PNG output
- optional release notes file if the README refresh needs to be referenced from the `v1.2.2` release note

The implementation should reuse existing patterns where possible rather than inventing a parallel README/gallery system.

## Content Strategy

The rewritten README should emphasize these messages:

- DeckDown is local and open source
- DeckDown is markdown-first and repo-native
- DeckDown ships a real Studio workflow on localhost
- DeckDown renders serious presentation features including Mermaid and LaTeX
- DeckDown outputs real deliverables, not just previews

The README should avoid sounding like generic AI tooling marketing. It should stay grounded in what the repo actually does today.

## Visual Strategy

The new deck and screenshots should use a warm editorial style:

- soft, paper-like background treatment
- restrained but confident accent color
- typography with contrast between display and body copy
- enough visual atmosphere to feel premium without obscuring readability

The deck should not look like a default corporate slide template or generic dark-mode AI demo.

## Verification

Implementation is complete only when all of the following are true:

- the new showcase deck source renders successfully
- the rendered PNG assets exist and look correct
- the rewritten README references the new visuals and paths correctly
- the deck can be opened locally in Studio
- localhost Studio screenshots are captured from the new deck workflow
- automated verification still passes:
  - `npm test -- --runInBand`
  - `npm run release-check`

## Risks

- README polish can drift into product claims that exceed the actual feature set
- Generated screenshots can become stale if taken from the wrong deck or wrong UI state
- A too-artistic hero slide could stop feeling like a DeckDown artifact and start feeling like marketing-only decoration

## Recommended Implementation Approach

1. Create the new showcase deck first so the README can be written against real assets.
2. Render the deck and validate the output locally.
3. Open the deck in Studio and capture new screenshots.
4. Rewrite `README.md` around the final rendered assets and screenshots.
5. Run full verification and leave the new deck available on localhost for review.
